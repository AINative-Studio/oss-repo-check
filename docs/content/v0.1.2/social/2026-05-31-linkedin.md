# LinkedIn Post: Announcing quaid-scanner

---

I have spent twenty years working in and on open source communities, and in that time I have been asked the same question in dozens of forms: _Is this project healthy?_
The question sounds straightforward.
The answers people produce for it are slow, inconsistent, and do not scale.
I have watched projects fail — not from technical debt but from what I think of as unexamined health debt: governance never written down, contributor funnels never cultivated, security posture assumed rather than verified, inclusive language practices adopted only after someone pointed out a problem.
I built quaid-scanner to make that debt visible, systematically, at machine speed.
I named it after myself — my GitHub handle is `quaid` — because I am putting my name on the choices inside it.

quaid-scanner is an open source TypeScript CLI that evaluates any repository — local path or GitHub URL — across six weighted pillars: security and supply chain (25%), governance and legal (20%), community health (15%), AI-native readiness (15%), inclusive language (15%), and technical rigor (10%).
It runs 43 scanners drawing on four frameworks I trust: the OpenSSF Scorecard project, CHAOSS metrics, The Open Source Way 2.0, and the Inclusive Naming Initiative.
It outputs a score from 0–10, a detailed JSON report, and a findings list where every entry includes a severity, a category, a suggestion for remediation, a data source label, and a reference URL linking to the authoritative standard for that check.
The people I have built it for are OSPOs evaluating dependency portfolios, foundation committees making incubation decisions, maintainers trying to understand where their project stands relative to best practice, and the agents that are increasingly doing all of that work on behalf of humans.

The design choice that differentiates quaid-scanner most is that it is built to be operated by an agent, not read by a human.
JSON is the primary output format; markdown is secondary.
Every finding has machine-parseable fields and a suggestion that is the raw material for a backlog story.
An MCP server ships with the tool, exposing `scan_repository` and `graph_query` as agent tools.
A Claude Code skill is in the repository.
The workflow I have validated on multiple portfolio scans: an agent scans a repository, parses findings by pillar and severity, fills a report template from the structured JSON, derives user stories from findings, and opens GitHub issues from those stories.
A human reviews at each step.
The heavy lifting is done.

The current release, v0.1.2, adds an evidence layer designed around auditability.
Every finding now carries a `dataSource` field (`api`, `local`, or `heuristic`) indicating where its data originated — a finding backed by a live ClearlyDefined.io API call is a different confidence level than one inferred from file structure.
Every scanner now populates a `referenceUrl` pointing to the authoritative source for that check: the OpenSSF Scorecard viewer for the specific project, the ClearlyDefined definition page for a specific dependency, the CHAOSS metric for a community health check.
Markdown reports end with a Score Rationale table showing each pillar's weight, raw score, and weighted contribution.
Two new scanners also ship in this release: a Naming Scanner that extends the Inclusive Naming Initiative check to the project's own identity (package name, README heading, git slug), and a ClearlyDefined License Cross-Validation scanner that checks production npm dependencies against the ClearlyDefined.io public API.

The name matters beyond the branding.
quaid-scanner represents my judgment about what OSS health means — grounded in four frameworks I trust but filtered through twenty years of experience evaluating open source projects.
I run inclusive language checks across source, documentation, and the project name itself because I believe the project's own identity matters.
I flag AI-native readiness because I think it is an underweighted signal of long-term health.
I cross-validate license data against ClearlyDefined because I think the project's own LICENSE file is not enough.
You might weight these differently.
You might know a check that belongs here and is missing.
The license is Apache-2.0 and the repository is public for exactly this reason: fork it your way, build your own opinionated scanner, and if a scanner or skill would be useful to others, open a pull request.
These are my scanners; I hope they are a starting point for yours.

The v0.1.3 roadmap addresses the next trust question: not just _where does this finding come from_ (which v0.1.2 answers) but _is this finding correct_.
Two mechanisms: a cross-validation harness that independently queries the OpenSSF Scorecard API and the `licensee` CLI and diffs the outputs against quaid's findings — failing CI if the discrepancy rate exceeds a configured threshold — and a ground-truth corpus of synthetic fixture repositories with precisely known properties, run through the full scanner on every `npm test`.
The mutation testing approach is the most useful part of the corpus: start from a "perfect repo" where every check passes, apply one controlled change, assert the scanner catches exactly the expected finding.
If the mutation test passes, the scanner does what it claims to do.

`npm install -g quaid-scanner`

The repository, the MCP server configuration, the Claude Code skill, and the full scanner reference are at github.com/quaid/quaid-scanner.
If you are in OSPO work, foundation governance, OSS sustainability research, or building agent workflows over code portfolios, I would value your feedback on what the scanner gets wrong — false positives, missing signals, pillars that need reweighting.
That feedback is the data v0.1.3 is built on.
