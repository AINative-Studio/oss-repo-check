# LinkedIn Post: quaid-scanner v0.1.2

---

I shipped quaid-scanner v0.1.2 today — a maintenance and feature release for the agent-first OSS repository health scanner I published last month.
The short version: the new release is about trust.
When a tool tells you that a dependency license is copyleft-incompatible, or that a project name contains a term flagged by the Inclusive Naming Initiative, the obvious follow-up question is _where is this coming from?_
v0.1.2 builds the answer to that question into the output.

The core addition is what I am calling the evidence layer.
Every finding now carries a `dataSource` field — `api` when the data is cross-validated against an external service, `local` when it is derived directly from repository contents, `heuristic` when it is inferred from structure or absence.
Every scanner now populates a `referenceUrl` pointing to the authoritative source for that check: the OpenSSF Scorecard viewer for security posture findings, the ClearlyDefined definition page for license data, the relevant CHAOSS metric for community health checks.
Markdown reports end with a Score Rationale table showing each pillar's weight, raw score, and weighted contribution to the overall score.
The score is no longer a number that appears from somewhere; it is the visible sum of transparent inputs.

Two new scanners bring the total to 43.
The Naming Scanner extends the Inclusive Naming Initiative check to the project's own identity — `package.json` name, README H1 heading, and git remote repository slug — not just its source and documentation.
If the project's name contains a flagged term, that is a CRITICAL finding with a rename suggestion, not a low-priority INFO.
The ClearlyDefined License Cross-Validation scanner cross-checks production npm dependencies against the ClearlyDefined.io public API, the most comprehensive database of community-verified open source licenses.
Copyleft dependencies, unrecognised SPDX expressions, and missing license data each produce findings with links to the specific ClearlyDefined definition pages.

A smaller but important practical addition: `.quaid-scanner-ignore` file support.
Placing a `.quaid-scanner-ignore` at the root of any scanned repository (same format as `.gitignore`) causes the inclusive language scanners to skip those paths.
This matters because test fixtures and documentation often contain flagged terms intentionally — as test assertions or as examples of what not to write.
Without a way to exclude those paths, every false positive in test code adds noise that erodes confidence in the signal findings.
This also activates the `excludePatterns` configuration option that existed in the type definitions since the initial release but was never connected to the scanner implementations.

quaid-scanner v0.1.2 is on npm now — `npm install -g quaid-scanner` — and the repository is at github.com/quaid/quaid-scanner.
I also took the time this release to make the tool meet its own standards: `CODE_OF_CONDUCT.md` (Contributor Covenant 2.1), `SECURITY.md`, and `SUPPORT.md` are committed; branch protection is on `main`; and GitHub issue templates are in place, including one specifically for false positives.
That last one matters because a false positive report is the feedback I most want — every confirmed false positive is either a tuning problem or a documentation gap.

v0.1.2 answers _where does this finding come from?_
v0.1.3 answers _is this finding correct?_ — and it does so via two complementary mechanisms.
The first is a cross-validation harness: a developer tool and weekly CI workflow that independently queries the OpenSSF Scorecard API and the `licensee` CLI (GitHub's own license detector) and diffs the results against quaid's findings.
Discrepancies above a configured threshold fail the workflow.
The second is a ground-truth corpus: a set of synthetic fixture repositories with precisely controlled properties, each run through the full orchestrator on every `npm test` run.
The most useful variant is mutation testing — start from a "perfect repo" where every check passes, apply a single change (remove a permissions block, pin a dependency to a mutable tag, add a flagged term to the package name), run the scanner, assert that exactly the expected finding appears.
If the mutation test passes, the scanner catches what it claims to catch.
If it does not, you have a regression before it reaches a user.

quaid-scanner is built on the collective work of the OpenSSF Scorecard project, the CHAOSS community, The Open Source Way project, and the Inclusive Naming Initiative.
I am trying to operationalize the best thinking I have found on what open source health actually means, structured so an agent can act on it and a human can audit the result.
If you are working on OSPO programs, foundation governance, OSS sustainability research, or agent workflows over code repositories, I would value your perspective on what the scanner gets wrong — what it misses, what it over-weights, where the map needs more detail.
