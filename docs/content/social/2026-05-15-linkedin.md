# LinkedIn Post: quaid-scanner v0.1.1

---

I just shipped quaid-scanner v0.1.1 to npm — an agent-first OSS repository health scanner that evaluates any public or private open source project across six weighted pillars: security, governance, community health, AI-native readiness, inclusive language, and technical rigor.
It runs 41 scanners in seconds, outputs structured JSON designed to be parsed by an agent rather than read by a human, and includes a Claude Code skill and MCP server so the full scan-to-backlog workflow can be driven end-to-end without a human in the loop.
I built this because I have spent twenty years watching open source projects accumulate invisible health debt — governance never written down, security posture assumed rather than verified, contributor funnels never cultivated — and I wanted a tool that made that debt visible at machine speed.

The _how_ is a plugin-based scanner architecture in TypeScript, each scanner implementing a common interface and registering against a pillar.
Point it at a local path or a GitHub URL; it clones the target if remote, runs the full scanner suite, and emits a scored JSON report with a `severity`, `category`, and `suggestion` on every finding.
The suggestion field is not decoration — it is the raw material for a user story, one the agent can draft, attach to a GitHub issue, and drop into a sprint backlog without further human translation.

What quaid-scanner actually surfaces is the gap between a project that _looks_ healthy and one that _is_ healthy.
I ran it against a public project this week that had an 85% issue closure rate, an active maintainer, and a welcoming score of 90/100 — genuine signals of care — alongside five GitHub Actions workflows with no `permissions:` block and thirteen action dependencies pinned to mutable version tags rather than commit SHAs.
The score was 1.7/10 overall.
The findings became a 14-story backlog in about three minutes.

The _why_ is that OSS health assessment does not scale manually.
Every OSPO evaluating a dependency portfolio, every foundation deciding which projects to incubate, every maintainer trying to understand where their project stands relative to best practice — they are all doing the same slow, inconsistent, human-labor-intensive work that a structured scanner can do faster and more consistently.
I also believe the AI-native readiness pillar is underweighted in existing tooling: whether a project has agentic rule files, a properly structured model card, or dataset provenance documentation is increasingly a signal of long-term health, not a niche concern.

quaid-scanner v0.1.1 is on npm now — `npm install -g quaid-scanner` — and the repository is at github.com/quaid/quaid-scanner.
Scan history persistence and trend analysis are already live: if you have ZeroDB credentials, every scan stores automatically and `alertOnDrop` catches regressions in CI.
Ecosystem intelligence (`--ecosystem` flag) is also live: rivals, partners, user communities, and strategic recommendations alongside health scores.
The roadmap is in the public PRD: accuracy validation via cross-validation against OpenSSF Scorecard and the `licensee` CLI is next, followed by a ground-truth corpus of synthetic repos with expected findings asserted on every test run.
If you are an OSPO, a foundation maintainer, or building agent workflows over OSS portfolios, I would genuinely value your feedback on what the scanner gets wrong — false positives, missing signals, pillars that need reweighting.

This tool exists because of the collective work of the OpenSSF Scorecard project, the CHAOSS community, The Open Source Way project, and the Inclusive Naming Initiative — frameworks that have spent years articulating what healthy open source actually looks like.
quaid-scanner is not an opinion about OSS health; it is an operationalization of the best thinking I have found on the subject, structured so an agent can act on it and a human can audit and refine the result.
If you work on OSS health, sustainability, or agentic tooling, I would love to hear where you think the map needs more detail.
